Design system — build order×
Why first: Every feature module imports from the design system. Building it first means you never retrofit consistency — you ship it.
1
CSS variables + Tailwind config
Define all color tokens, spacing scale, radius vars in tailwind.config.ts. Dark mode via class strategy. This is the single source of truth for every color in the app.
2
shadcn/ui component install + theme
Install and theme: Button, Input, Select, Dialog, Sheet, Dropdown, Tooltip, Badge, Card, Table, Skeleton, Tabs. Override default Radix styles to match Synalytix tokens.
3
Layout primitives
Shell, Sidebar (collapsible + icon mode), TopNav (org switcher, search trigger, bell), PageContainer (max-width, breadcrumbs). These wrap every page.
4
Data display components
DataTable (virtualized, sortable, filterable), ChartCard, MetricBadge, EmptyState (contextual illustration + CTA), ErrorBoundary, SkeletonCard. These appear on every dashboard page.
5
Storybook (optional but recommended)
Document each component with light/dark variants, loading/error/empty states. Becomes the team's component reference. Deploy to Chromatic for visual regression testing.
Next.js app — feature build order×
Routing strategy: (marketing) → /app/(auth)/login → /app/(dashboard)/[orgSlug]/... — separate route groups keep auth and app layouts isolated. Server Components fetch data; Client Components handle interactivity.
1
Auth flow
Login, signup, forgot password, OAuth callbacks (Google, GitHub) via Supabase Auth helpers for Next.js. Middleware redirects unauthenticated users. useAuth hook. Onboarding flow (create org → invite team).
2
App shell + navigation
Sidebar with nav items, org switcher, Cmd+K global search (cmdk library), notification bell with unread count from Supabase Realtime. Settings drawer (not page). Responsive collapse behavior.
3
Dataset module
Upload flow (drag-drop → signed URL → progress → processing status polling), dataset list, schema inspector, data preview table (virtualized), query builder UI.
4
Dashboard builder
Grid layout (react-grid-layout), widget palette, widget config panel, chart types (line, bar, pie, metric, table). Save layout. Public share view (no auth required).
5
AI features UI
NLQ chat interface (streaming responses), insights feed, anomaly alerts, report summarization trigger + result display. All async — skeleton while job processes.
6
Team + billing UI
Members table, invite modal, role management. Billing page: current plan, usage meters, upgrade CTA → Stripe Checkout redirect. Invoice history table.
Dashboard builder — the hardest UI to get right×
Complexity warning: The dashboard builder is the most complex UI in the app. Allocate 3–4 weeks minimum. Plan it before building — grid system, widget state, and query binding must be designed together.
1
Grid system (react-grid-layout)
Responsive 12-column grid. Drag to reposition, resize handles. Layout JSON serialized to backend on save. Distinguish edit mode vs view mode clearly.
2
Widget registry pattern
Each widget type (LineChart, BarChart, MetricCard, DataTable, TextBlock) is a self-contained component that receives config_json + executes its own query. Widget registry maps type → component.
3
Query binding UI
Each widget has a query builder panel: pick dataset, pick columns, add filters, group-by, aggregation. Preview updates live (debounced 500ms). This is the core product value.
4
NLQ widget mode
Toggle from query builder → natural language input. Send to /nlq endpoint, stream back SQL + explanation, render result in widget. Show generated SQL in collapsible panel for transparency.
5
Dashboard filters + cross-filtering
Global date range picker. Clicking a bar in chart A filters chart B. Requires a shared filter context (Zustand store) that widgets subscribe to. V2 feature — ship after core builder works.
Frontend ↔ backend integration×
1
API client setup
src/lib/api.ts — typed fetch wrapper that attaches Supabase JWT, handles 401 refresh, returns typed responses. Generate TypeScript types from OpenAPI spec (openapi-typescript).
2
TanStack Query hooks
One query hook per resource: useOrganization, useProjects, useDatasets, useDashboard, useInsights. Optimistic mutations for fast UI. Invalidation strategy per entity graph.
3
Supabase Realtime subscriptions
useRealtimeInsights hook subscribes to ai_insights channel. useJobStatus polls or subscribes to dataset processing updates. Drives live notification bell without polling.
4
Error handling UX
API errors → toast notifications (Sonner). 402 plan limit errors → upgrade modal. 401 → redirect to login. Network errors → retry with exponential backoff. Never expose raw error messages to users.
E2E testing with Playwright×
1
Critical path: signup → onboarding
New user signs up via email, verifies (Supabase dev mode bypasses email), creates org, completes onboarding checklist. Must pass before every deploy.
2
Dataset upload + processing
Upload a CSV fixture → assert processing completes → assert schema detected correctly → assert preview table renders. Tests the full async job pipeline end-to-end.
3
Dashboard create + share
Create dashboard → add chart widget → bind to dataset → save → generate public link → open in incognito → assert chart renders without auth.
4
Billing upgrade flow
Hit a plan limit → see upgrade prompt → redirect to Stripe Checkout (test mode) → webhook fires → plan upgrades → feature now accessible. Uses Stripe CLI for local webhook testing.
Closed alpha — how to run it right×
The alpha phase determines product-market fit. Ship rough, learn fast. The backend and UI you built are 90% useless until real users hit real problems with their real data.
1
Find 5–10 design partners
Target companies already doing analytics manually (spreadsheets, Metabase, Looker). Offer free access for 3 months in exchange for weekly 30-min feedback calls. No cold outreach — warm intros only.
2
Instrument everything with PostHog
Track: time to first dataset upload, time to first dashboard created, NLQ queries per session, features used per plan. You need data to know what to build next — not opinions.
3
Set up a feedback channel
Slack Connect or Discord with alpha users. Respond to every bug report within 4 hours. Ship fixes same day when possible. Speed of response is the product at this stage.
4
Weekly changelog
Every Friday, email alpha users what shipped. Makes them feel heard, builds habit, and primes them to become advocates at launch. Use a simple markdown template.
Marketing site — what to build×
1
Landing page
Hero with demo video or interactive preview. 3 core value props. Customer logos (alpha partners). Pricing table. CTA to waitlist or sign up. Build with Next.js or Framer — not WordPress.
2
Docs site
Mintlify or Docusaurus. API reference auto-generated from OpenAPI spec. Quickstart guide, connector docs, webhook reference. Good docs reduce support load by 60%.
3
Changelog
Public changelog at /changelog or changelog.synalytix.com. Ship weekly. Links from in-app notification bell. Shows momentum to prospects and keeps existing users engaged.
4
SEO + blog
Target "analytics for [industry]" keywords. 2–3 posts/month on data topics your ICP Googles. Long tail content compounds over 6–12 months into meaningful organic traffic.
Public launch checklist×
1
Switch Stripe to live keys
Test every billing flow in staging with live keys first. Ensure webhook signing secrets are rotated. Confirm Stripe Tax is configured for your target markets.
2
Status page + uptime monitoring
Betterstack or Instatus. Monitor: API health endpoint, Supabase, Railway app. Set up PagerDuty or similar for on-call alerts. SLA requires you to know about outages before users tweet.
3
Product Hunt launch
Schedule for Tuesday–Thursday. Prepare hunter, gallery assets, tagline, first comment. Brief alpha users to upvote and comment on launch day. Aim for top 5 of the day.
4
Support infrastructure
Intercom or Plain for in-app chat. Help docs linked from every error state. Define SLA: free = community, pro = 24h, business = 4h. Brief first responders before launch.
Observability & analytics stack×
1
PostHog product analytics
Session recordings on key flows (upload, dashboard builder). Funnels: signup → first dataset → first dashboard (where do users drop off?). Feature flags for gradual rollouts.
2
Sentry performance monitoring
Already configured in Phase 8. Set p95 latency alerts (> 2s = page, > 5s = incident). Track slow queries. Error rate alerts by route prefix (/api/v1/ai is allowed to be slower than /api/v1/datasets).
3
Business metrics dashboard
Internal Synalytix dashboard tracking: MRR, churn rate, DAU/MAU, avg datasets per org, NLQ query volume, p99 API latency. Eat your own cooking — this validates the product.
Enterprise track — when to start×
Don't start enterprise track until you have 3+ inbound enterprise requests. Building SSO and SOC 2 before demand wastes 3+ months that should go toward user growth.
1
SSO / SAML
WorkOS or Auth0 for SAML/OIDC. Adds Okta, Azure AD, Google Workspace SSO. 2–3 week integration. Triggers for 1st inbound enterprise request that mentions SSO as a blocker.
2
SOC 2 Type I audit
Vanta or Drata automates 60–70% of evidence collection. ~3 months to prepare, ~1 month for auditor. Type I first (point-in-time), then Type II (6-month observation period) for enterprise deals.
3
Custom contracts + SLAs
99.9% uptime SLA, DPA for GDPR, MSA template, security questionnaire boilerplate. Use a lawyer for the first contract — don't template-hunt on GitHub.
4
Private cloud / VPC deployment
Some enterprises require data to never leave their VPC. Prepare a Docker Compose + Helm chart self-hosted deployment path. Supabase self-hosted is an option here.
