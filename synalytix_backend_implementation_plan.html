Phase 2 Prisma schema, migrations & Supabase setup Day 2–3 ›
Architecture note: Prisma connects via Supabase connection pooler (port 6543, pgBouncer mode) for the app. Direct connection (port 5432) is used only for migrations. RLS is enabled on all tenant tables as a defense-in-depth layer — RBAC middleware is the primary enforcement.

Full Prisma schema — all 14 core tables
users, organizations, organization_members, projects, datasets, dashboards, widgets, reports, ai_insights, workflows, activity_logs, api_keys, billing_events, invitations. All tenant tables carry org_id. Soft deletes via deleted_at on mutable entities.
Prisma14 models
Indexes strategy
FKs indexed. Composite (org_id, created_at) on all tenant tables. Unique on users.email, api_keys.hashed_key. GIN indexes on JSONB cols (layout_json, config_json, schema_json). Partial index on active subscriptions. Raw SQL migration for GIN (Prisma limitation).

SQL migration enabling RLS on every tenant table. Policy: authenticated users can only SELECT/UPDATE/DELETE rows where org_id matches their JWT claim. Service role key bypasses RLS for backend-to-backend operations.

Phase 3 Middleware stack & auth layer Day 3–4 ›
Auth flow: Supabase Auth issues JWTs. Every request hits our Express server with Authorization: Bearer <token>. Our middleware verifies the token with Supabase's JWKS endpoint (cached), extracts user + org context, then enforces RBAC. We never issue our own JWTs.
Middleware order (applied in this sequence)
1. Request ID + structured logger
Attach x-request-id (nanoid) to every request. Winston logger emits JSON in production (Railway log drain), pretty-print in dev. Log: method, path, status, duration, request_id. Never log Authorization headers or passwords.
WinstonMorgan
2. Security headers
Helmet.js with tight CSP. CORS — strict origin whitelist from env (FRONTEND_URL). No wildcard origins in production. SameSite cookie policy. X-Frame-Options DENY.
HelmetCORS
3. Rate limiter (Redis-backed)
Global: 200 req/min per IP. Auth endpoints: 10 req/min per IP (brute-force protection). AI endpoints: 30 req/min per user. Storage in Redis with sliding window. Returns 429 with Retry-After header.
rate-limit-redisRedis
4. JWT authentication middleware
Verifies Supabase JWT via JWKS (public key cached 1hr in Redis). Extracts user_id, email, org_id from claims. Attaches to req.user. Returns 401 on invalid/expired token. Separate API key auth path for programmatic access.
Supabase AuthSecurity
5. RBAC authorization middleware
requireRole(...roles) factory. Checks organization_members for user's role in current org. Role hierarchy: Owner → Admin → Manager → Member → Viewer. Caches role lookups 5min in Redis. Returns 403 on insufficient permissions.
RBACRedis cache
6. Global error handler
Catches all thrown errors. Maps Zod ValidationErrors → 400, AuthError → 401, ForbiddenError → 403, NotFoundError → 404, ConflictError → 409. All others → 500. Standard envelope: {success, error: {code, message, details?}}. Sentry capture on 5xx.
Error boundarySentry

Phase 4 Core REST API — org, project, team, API keys Day 4–6 ›
Pattern: Every route module = router.ts (routes + validation) + controller.ts (req/res handling) + service.ts (business logic + DB). Controllers never touch Prisma directly. Services never touch req/res.
Route modules

POST /orgs (create), GET /orgs/:id, PATCH /orgs/:id (name, slug), DELETE /orgs/:id (owner only, soft delete). GET /orgs/:id/usage — returns seat count, dataset rows, AI query count against plan limits. OrgService handles slug uniqueness, plan enforcement.

Phase 6 AI features — NLQ, insights, anomaly detection Day 8–10 ›
Safety model: All AI-generated SQL runs against a read-only Postgres role. Generated queries are validated (no DDL/DML, complexity check) before execution. Insights below 0.7 confidence threshold are discarded. All AI ops are async — client polls or listens via Supabase Realtime.
Natural language query (NLQ) endpoint
POST /datasets/:id/nlq — accepts natural language question. NlqService injects dataset schema into Anthropic prompt, requests SQL + explanation. Validates generated SQL (read-only, max 10 joins, no subquery bombs). Executes via QueryBuilder. Returns {sql, explanation, results}. Metered per org.
Anthropic APISQL validationUsage metered
AI insights generation job
BullMQ job (insightQueue) scheduled daily per active dataset. InsightProcessor computes statistical summary (min/max/mean/stddev per column, null rates, top values). Sends to Anthropic for narrative insights. Stores in ai_insights with confidence_score. Discards score < 0.7.
BullMQ cronAnthropic
Anomaly detection service
AnomalyService runs on new data ingestion. Z-score based threshold detection (|z| > 3 = anomaly) per numeric column. Flagged anomalies sent to Anthropic for plain-English explanation. Stored as ai_insights of type ANOMALY. Triggers alert workflow if configured.
StatisticalLLM explanation
Report summarization
POST /reports/:id/summarize — async job. ReportSummaryProcessor fetches report content + referenced dataset stats. RAG-style: chunks content, retrieves relevant data points, sends to Anthropic with citation prompt. Returns structured summary with data citations.
RAG patternCitations
AI insights API
GET /projects/:id/insights — paginated list with filters (type, dataset, date, confidence). GET /insights/:id — full insight with metadata. POST /insights/:id/dismiss. Supabase Realtime broadcast when new insight is written (client subscribes by org_id channel).
Realtime4 endpoints
